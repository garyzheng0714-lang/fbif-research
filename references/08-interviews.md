# 模块八：深度采访与深度文章

## ⚠️⚠️⚠️ 字数红线：≥30,000字（最重模块！）。不到30,000字 = 严重不达标，必须返回补充web_fetch。不到10,000字 = 说明完全没有执行web_fetch。

## ⚠️⚠️⚠️ 最重要的模块——必须严格遵守

本模块是整个调研中**字数最多、最重要**的模块。用户的核心需求是获取第一手的完整素材。

**本模块的目标体量：3-10万字。** 如果只有一两千字，那是严重违规。

---

## 三条绝对铁律（违反任何一条 = 调研失败）

### 铁律一：原文必须完整保留

web_fetch获取到的文章完整正文，**一字不改，全文录入HTML**。

- 一篇深度采访原文通常有3000-10000字
- 必须把这3000-10000字**全部**放进HTML
- 不能只取"关键段落"
- 不能用自己的话"概括"
- 不能因为"太长"而截断

### 铁律二：中文翻译必须完整

在保留原文全文的基础上，提供**完整的逐段中文翻译**。

- 原文有多少段，翻译就有多少段
- 翻译要地道，不要翻译腔
- 如果原文是中文，不用翻译

### 铁律三：禁止用摘要代替全文

| 违规行为 | 正确做法 |
|----------|----------|
| 用web_search摘要的几句话当"原文" | web_fetch获取完整正文，全文录入 |
| 只录入"关键段落摘要" | 录入全文，一段不省 |
| 用自己的话概括文章并称"原文" | 保留原文，另外提供翻译 |
| 只有一两千字就标记完成 | 必须达到3万字以上 |
| 声称"篇幅限制"截断 | HTML没有篇幅限制，全文放入 |
| 只列文章标题和链接 | 必须放完整正文+完整翻译 |

---

## 正确的执行流程（每篇文章都要走完）

```
步骤1：web_search → 找到文章URL
步骤2：web_fetch URL → 获取完整正文（通常3000-10000字）
步骤3：将web_fetch返回的完整原文 → 一字不改录入HTML
步骤4：在原文下方 → 提供完整中文翻译（同样3000-10000字）

一篇文章在HTML中应占 6000-20000字（原文+翻译）
5-15篇文章 × 每篇6000-20000字 = 3-10万字
```

### 体量自检

| 检查项 | 红线 |
|--------|------|
| 单篇文章（原文+翻译） | 不应少于3000字（除非原文本身很短） |
| 模块总体量 | **不少于3万字** |
| 如果总体量不到1万字 | 说明没有执行web_fetch，**必须返回重做** |
| 如果总体量只有一两千字 | **严重违规，必须立即重做** |

## 素材类型与价值排序

**核心原则：先找本人的声音，再找别人的分析。**

| 优先级 | 素材类型 | 说明 | 价值 |
|--------|----------|------|------|
| ★★★★★ | 演讲/Keynote | 创始人/CEO在行业会议、TED、大学的公开演讲 | 最高：本人亲口、未经编辑的完整思路 |
| ★★★★ | 采访/对话 | 媒体专访、播客对话、视频访谈 | 本人声音，但经记者引导和编辑 |
| ★★★ | 创始人著作 | 书籍、专栏文章、公开信 | 本人亲笔，但是深思熟虑的"官方叙事" |
| ★★ | 深度报道 | 权威媒体的深度调查报道 | 记者视角，有独立采访和事实核查 |
| ★ | 第三方分析 | 行业分析师、咨询公司、学术论文 | 最远离一手信息，但可能有独特数据和框架 |

**执行要求：** 第一手素材（演讲+采访+著作）占比应超过50%。如果搜完发现全是第三方分析，必须回头补搜演讲和采访。

## 调研内容

### A. 深度采访文章

**核心采访对象**：创始人、CEO、品牌/产品的关键负责人/关键人物

**调研语言优先级**：
1. 品牌母国语言原文采访（如日本品牌找日文采访，德国品牌找德文采访）
2. 英文采访
3. 中文采访

**每篇采访必须保留的完整信息**：

| 信息项 | 要求 |
|--------|------|
| 文章标题 | 原文标题 + 中文翻译 |
| 采访平台/媒体 | 平台或媒体名称 |
| 采访时间 | 尽量精确到月份 |
| 采访人/作者 | 采访记者或作者姓名 |
| 被采访人 | 姓名 + 职位 |
| **原文全文** | **web_fetch获取的完整内容，一字不改，可能有数千字到上万字** |
| **中文翻译全文** | **逐段完整翻译，地道表达，字数与原文相当** |
| 原文链接 | 完整URL |

### B. 创始人/高层演讲与Keynote

**这是最高价值的素材类型，必须主动搜索。**

搜索关键词：
- `{创始人名} + speech / keynote / lecture / presentation / 講演 / Vortrag / conférence`
- `{创始人名} + TED / TEDx`
- `{品牌名} + keynote / conference speech / summit`
- `{创始人名} + {行业会议名} + speech`

搜索来源：行业会议/展会、TED/TEDx、大学客座演讲、品牌官方活动、投资者日/年会

**每篇演讲必须保留的信息**：演讲标题（原文+中文）、演讲场合、时间、演讲人（姓名+职位）、**演讲全文/文本**（web_fetch或视频提取的完整内容）、**中文翻译全文**、来源链接、时长（如为视频）

### C. 深度视频采访

来源：YouTube、品牌官方视频、行业会议、播客等

**⚠️ 视频和播客是深度采访的重要来源，不能因为是视频/音频就跳过。必须尽一切努力提取文本内容。**

处理方式（按优先级依次尝试，每一步都要实际执行）：

1. **搜索现成transcript** — 很多视频/播客已经有别人整理的文字版
   - 搜索 `"{视频标题}" transcript`
   - 搜索 `"{播客名}" "{嘉宾名}" transcript`
   - 搜索 `"{演讲标题}" full text`
   - 行业会议官网通常发布演讲文字版
   
2. **YouTube字幕提取** — YouTube大部分视频有自动字幕或手动字幕
   - web_fetch视频页面获取描述和基本信息
   - 搜索 `youtube transcript "{video_id}"` 或 `youtube subtitles "{视频标题}"`
   - 尝试第三方字幕提取网站（搜索 "youtube transcript downloader"）
   - YouTube的自动字幕虽然可能有小错误，但完整保留比没有好得多

3. **播客平台transcript** — 
   - web_fetch播客官网页面（很多播客在官网提供完整transcript）
   - Apple Podcasts / Spotify 页面有时嵌入文字内容
   - 搜索 `"{播客名}" show notes "{期数}"`
   - 很多播客使用Otter.ai/Rev生成transcript并发布

4. **文字版搜索扩展** —
   - 搜索作者/嘉宾是否在LinkedIn/Medium/个人博客发布了相同内容的文字版
   - 搜索行业媒体是否对该视频/播客做了文字报道
   - 搜索 `"{嘉宾名}" "{关键话题}" article` — 嘉宾可能在其他渠道讲过同样内容

5. **以上全部失败时** — 必须仍然保留完整信息：
   - 视频标题（原文+中文）、平台、发布时间、时长
   - 演讲人/被采访人、完整链接URL
   - 简要内容描述（标注"编者概述，非原文"）
   - **写入 `blockers.md` 和来源清单，聊天中只做简短提示**

**提取的transcript处理规则：**
- transcript同样需要：**完整保留原文 + 提供完整中文翻译**
- 标注来源类型：`[来源：YouTube字幕提取]` / `[来源：播客官网transcript]` / `[来源：第三方transcript]`
- 字幕中的时间戳保留（方便用户回溯视频原文）
- 口语化的transcript不需要"修改"成书面语——保留原始口语风格，翻译时使其通顺即可

### D. 深度文章（非采访类）

行业深度文章：关于该品牌、产品、或品类主题的深度分析文章。

每篇文章必须保留：文章标题（原文+中文）、作者、发布平台、发布时间、**原文全文**（web_fetch获取的完整正文）、**中文翻译全文**、原文链接

### E. 创始人著作

主动搜索创始人/CEO是否出过书或有专栏：
- `{创始人名} book / 著書 / Buch / livre`
- `{创始人名} author / 著者`

如找到书籍，记录：书名、出版年份、核心主题，并搜索书评/摘要

## 搜索策略

### 第一步：广泛搜索，建立文章URL清单

以品牌母国语言为主，构建搜索关键词矩阵：

**采访类**：
- `{品牌名} + インタビュー / interview / 采访 / Interview（德语）`
- `{品牌名} + {创始人名} + interview`
- `{品牌名} + CEO + 対談 / conversation / Gespräch`
- `{品牌名} + 開発秘話 / behind the story / 背后的故事 / Geschichte`
- `{品牌名} + 成功の秘密 / secret of success / Erfolgsgeschichte`
- `{品牌名} + 失敗 / failure / 失败 / Krise`
- `{品牌名} + ブランドストーリー / brand story / Markengeschichte`
- `{品牌名} + profile / portrait / Porträt`
- `{品牌名} + founder story / Gründer`

**演讲/Keynote类**（必须单独搜索，不能遗漏）：
- `{创始人名} + speech / keynote / lecture / 講演 / Vortrag`
- `{创始人名} + TED / TEDx`
- `{品牌名} + keynote / conference speech / summit presentation`

**创始人著作类**：
- `{创始人名} + book / author / 著書 / Buch`
- `{创始人名} + column / essay / 寄稿`

### 第二步：逐个通过 Jina Reader 获取全文

**这是本模块的核心步骤。** Jina Reader (`https://r.jina.ai/{url}`) 能将任何网页转为干净 Markdown，是最可靠的全文获取方式。

对搜索结果中每一篇有价值的文章：

```
1. WebFetch URL: https://r.jina.ai/{article_url}
   Prompt: "Extract the complete article text. Return every paragraph. Do not summarize."
2. 验证返回字数（正常文章 ≥1000字，深度采访 ≥3000字）
3. 完整原文一字不改写入 sources/src-{xxxx}.md
4. 同一文件追加完整中文翻译（逐段对照翻译）
5. 整理后追加到 modules/m8_interviews.md
6. 更新 source-inventory.json（status: fetched, word_count: xxx）
```

**字数验证是防止内容缺失的关键。** 如果返回内容明显不完整，必须走回退链路。

## 抓取失败时的7级回退链路

**用户需要这些原始资料做调研和二次创作，抓取失败不等于可以跳过。**

1. **Jina Reader 重试** — 偶尔因超时失败，直接重试
2. **直接 WebFetch 原URL** — 有些站点直接抓取效果更好
3. **搜索转载/镜像** — `"{文章标题}"` 找其他平台的全文转载
4. **Wayback Machine** — `WebFetch URL: https://r.jina.ai/https://web.archive.org/web/{原URL}`
5. **多次搜索拼接** — 用文章中独特短语、人名+关键词做5-10次搜索拼接80-95%内容
6. **PDF版本** — `"{文章标题}" filetype:pdf`
7. **作者其他渠道** — LinkedIn/Medium/个人博客可能有同内容

视频/播客参照上方"C. 深度视频采访"的完整提取流程。

**只有7级全部实际尝试并失败后，才能列入"待用户手动获取"清单。**

**所有方法都失败后的最终处理：**

不能直接跳过。必须同时做两件事：

**第一件：HTML中保留完整信息表。** 在"未能获取全文"汇总区录入该文章的全部元信息（标题、平台、链接等），以及已通过搜索获取到的部分内容，确保不丢失任何线索。

**第二件：在阻塞文件中单独列出待获取列表，并在聊天中仅给出路径提示：**

```
以下文章我尝试了所有自动获取方法均未成功，需要你手动提供内容：

| # | 原文标题 | 中文标题 | 平台 | 链接 | 已尝试的方法 |
|---|----------|----------|------|------|-------------|
| 1 | xxx | xxx | xxx | URL | 直接fetch/转载/存档/拼接均失败 |
```

## HTML中的展示格式

每篇访谈/文章用**卡片式布局**（使用 `.itw-card` 样式），中文标题最突出：

```html
<div class="itw-card">
  <div class="itw-head">
    <div class="itw-title">访谈：创始人XXX谈品牌创立初心</div>
    <div class="itw-meta">
      <span class="lbl">原文标题</span><span>Original Title Here...</span>
      <span class="lbl">平台</span><span>媒体名称</span>
      <span class="lbl">时间</span><span>2024年6月12日</span>
      <span class="lbl">被采访人</span><span>Name, 职位</span>
      <span class="lbl">链接</span><span><a href="URL">URL</a></span>
    </div>
  </div>
  <div class="itw-body">
    <h4>原文全文</h4>
    <p>（web_fetch获取的完整原文，一字不改，可能有数千字...）</p>
    <p>（第二段...）</p>
    <p>（第三段...）</p>
    <!-- 完整原文的每一段都要在这里 -->
    
    <h4>中文翻译</h4>
    <p>（完整的逐段中文翻译，字数与原文相当...）</p>
    <p>（第二段翻译...）</p>
    <p>（第三段翻译...）</p>
    <!-- 完整翻译的每一段都要在这里 -->
  </div>
</div>
```

**要求：**
- 中文标题最醒目（大字号+加粗），用文章内容概括
- 元信息分行展示，用grid对齐
- **原文全文和中文翻译全文必须完整呈现，不截断，不省略**
- 每篇文章的原文+翻译可能占6000-20000字，这是正常的

## 质量自检清单

1. ☐ **是否主动搜索了创始人/CEO的演讲和Keynote？**
2. ☐ 是否优先搜索了品牌母国语言的采访？
3. ☐ **是否对每篇文章都执行了web_fetch？**（不是只用搜索摘要）
4. ☐ **每篇采访的原文是否真正完整保留？**（不是摘要、不是概括）
5. ☐ **原文和中文翻译是否同时保留？**（翻译不能替代原文，两者都在HTML中）
6. ☐ **中文翻译是否逐段完整且地道？**
7. ☐ **视频/播客采访是否尝试了全部transcript提取方法？**（不能因为是视频就跳过）
8. ☐ 无法提取的视频/播客是否保留了完整元信息？
9. ☐ web_fetch失败的文章是否穷尽了所有补救手段？（重试/转载/存档/拼接/PDF）
10. ☐ 所有元信息（标题、平台、时间、人物）是否齐全？
11. ☐ 是否搜索了创始人著作？
12. ☐ **log_fetch验证报告是否所有URL保留率≥80%？**
13. ☐ **文档总字数是否达到3万字以上？**
14. ☐ **如果不到1万字 → 没有执行web_fetch → 必须返回补充**
15. ☐ **如果只有一两千字 → 严重违规 → 必须重做**
16. ☐ 第一手素材占比是否超过50%？
